# プロダクトをリポジションする — 「理解負債」を主役に、コード負債を優先順位信号へ（ナラティブ/ポジショニング）

## 概要
本プロダクトはコード負債と理解負債の両方に取り組んできたが、**コード負債（技術負債）の検知・可視化は
CodeScene / SonarQube / Code Climate 等の既存プロダクトで成熟しており差別化が弱い**。一方、本プロダクトの
真の強みは「**理解度を git から推測するのでなく、クイズで能動的に実測し、学習プランで返済する閉ループ**」に
ある。本 issue は、プロダクトの**物語・ポジショニング・コピー・ブランディング**を「**理解負債（Knowledge/
Understanding Debt）中心**」へ再構成する（コード負債は捨てず、後述のとおり「優先順位づけの信号」へ格下げ）。

> 本 issue は**ナラティブとドキュメント**の改稿に絞る。プロダクト UI / API の再構成は **059** が担う。
> コード負債の検知パイプライン（028）・二軸マトリクスは**維持**する（消さない）。

## 背景・目的

### 差別化の所在（なぜ理解負債を主役にするか）

| 既存プロダクト（CodeScene 等） | 本プロダクト（リポジション後） |
|---|---|
| コード負債 + git からの**知識偏り推測**（bus factor / knowledge map / オフボーディングリスク） | 理解度を**クイズで実測**（blame 非依存。単独開発・コードを書かない PM も計測可） |
| 偏りを**可視化して終わり**（受動的） | **学習プラン → クイズ再受験で返済する閉ループ**（能動的） |
| 「最後に触った人」を理解の代理指標にする | 「実際に理解しているか」を出題で確かめる |

- 「理解負債に取り組む」こと自体は CodeScene 等も（受動的に）やっており差別化にならない。**差別化は
  「能動的に測って、返済する」点**にある。052〜057（クイズ実測 KC・blame 非依存・機能単位）は既にこの線上。
- コード負債を完全に消すと、本プロダクトの主張だった二軸の優先順位（「危険なコード × 誰も理解していない = P0」）が
  失われ「ただの理解度クイズツール」に弱体化する。よって**コード負債は“理解負債を優先順位づけする信号”として残す**。

### 目的
1. プロダクトの一言ポジショニング（タグライン）を「理解負債を実測し返済する」に確定する。
2. README / ピッチ / 仕様書 / アプリ内コピーを、理解負債を主役・コード負債を脇役（信号）として書き換える。
3. 「Tech Debt Twin Agent」という副題（コード負債前提の語り）を見直す（後述の要決定）。

## 要決定（本 issue 内で確定させる）

- [ ] **副題 / タグライン**: 現 `Tech Debt Twin Agent`（`CLAUDE.md` / ブランディング）は技術負債前提。候補：
      「Knowledge Debt Twin」「理解負債を実測し返済するエージェント」等。プロダクト名 `DevDebtOps`（言語＝理解の
      メタファー、Galaxy/Rマーク）は理解負債とも整合するため維持を推奨。**最終決定はオーナー判断**。
- [ ] **コード負債の呼称**: プロダクト内で「コード負債」を“競合機能”でなく“リスク信号（hotspot lens）”として
      語る用語を統一（例：「危険度シグナル」「ホットスポット」）。
