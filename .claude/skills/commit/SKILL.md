---
name: commit
description: 適切な規約的コミットフォーマットと絵文字プレフィックスでgitコミットを作成
---

ユーザー入力はエージェントから直接提供されるか、コマンド引数として提供される可能性があります - プロンプトを進める前に必ず考慮してください（空でない場合）。

ユーザー入力:

$ARGUMENTS

以下の厳密な規約に従ってgitコミットを作成してください:

## コミットメッセージフォーマット

**必須フォーマット**: `<emoji> [<type>] <説明>`

### タイプ接頭辞（必ず一つ使用）:
- `:books:` **[docs]** - ドキュメントの変更、README更新、コメント
- `:sparkles:` **[feat]** - 新機能や新しい機能
- `:zap:` **[update]** - 既存機能の改善
- `:bug:` **[fix]** - バグ修正
- `:hammer:` **[chore]** - リファクタリング、依存関係更新、ビルド変更、その他

## 実行手順

1. **リクエスト解析**: `$ARGUMENTS` のユーザー説明を解析して理解する:
   - どのファイル/変更をコミットする必要があるか
   - 変更の性質（コミットタイプを決定するため）
   - 特定ファイルが言及されているか、すべての変更が含まれるべきか

2. **git status確認**: `git status` を実行して修正済み/追跡外ファイルを確認

3. **コミットタイプを決定** 変更に基づいて:
   - ファイル拡張子とパスをチェック（例：`.md` ファイル → docs）
   - キーワードの説明を分析（例："fix"、"add"、"update"、"refactor"）
   - 曖昧な場合、`git diff` で実際の変更を確認

4. **適切なファイルをステージ**:
   - 特定ファイルが言及されている場合: `git add <特定ファイル>`
   - "すべて"が言及されているか一般的な説明の場合: どのファイルかユーザーに確認
   - 確認なしにすべてのファイルを自動追加しない

5. **コミットメッセージ作成**:
   - 適切な絵文字とタイプで開始
   - 簡潔で明確な説明を記述（命令法）
   - 複数の変更がある場合は箇条書きを追加
   - 変更の意図が明確になるよう説明を具体的に記述する

6. **heredocフォーマットを使用してコミット作成**:
   ```bash
   git commit -m "$(cat <<'EOF'
   <emoji> [<type>] <説明>

   - <詳細 1>
   - <詳細 2>

   🤖 Generated with [Claude Code](https://claude.ai/code)

   Co-Authored-By: Claude <noreply@anthropic.com>
   EOF
   )"
   ```

7. **コミット確認**: `git show --stat HEAD` でコミットを表示

## 例

ユーザー: `/commit 規約とテンプレートを更新`
結果: `:books: [docs] 規約とテンプレートを更新`

ユーザー: `/commit ログインのバグを修正`
結果: `:bug: [fix] ログイン認証の問題を修正`

ユーザー: `/commit ユーザープロフィール機能を追加`
結果: `:sparkles: [feat] ユーザープロフィール機能を追加`

ユーザー: `/commit データベース接続をリファクタリング`
結果: `:hammer: [chore] データベース接続ロジックをリファクタリング`

## 重要な注意事項

- ステージングする前に常にコミット内容を確認
- 変更の意図が伝わる具体的な説明を使う
- 現在時制、命令法を使用（「追加」、「追加した」や「追加中」ではない）
- 最初の行は72文字以下に保つ
- 複雑な変更の場合はコミット本文に詳細を追加
- 機密情報やクレデンシャルは絶対にコミットしない
- プリコミットフックでコード品質を保証 - 可能な場合はスキップではなく問題を修正
- **コミットメッセージは必ず日本語で作成すること**（絵文字プレフィックスと[type]タグは英語のまま）
- 変更が多い場合は、複数のコミットに適切な粒度で分けること
