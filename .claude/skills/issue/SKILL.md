---
name: issue
description: GitHub issue を作成する。タイトル・本文・ラベルをインタラクティブに確認してから作成。
---

ユーザー入力はエージェントから直接提供されるか、コマンド引数として提供される可能性があります - プロンプトを進める前に必ず考慮してください（空でない場合）。

ユーザー入力:

$ARGUMENTS

以下の手順に従って GitHub issue を作成してください。

## issue タイトルフォーマット

**フォーマット**: `<説明>`

- 何をするか / 何が問題かを簡潔に記述
- 命令法（「〜を追加する」「〜を修正する」）
- 72文字以下

### 例

- `ダッシュボードの二軸マトリクスを実装する`
- `Code Debt Agent の重複コード検知ロジックを追加する`
- `GitHub Webhook の受信エンドポイントを作成する`

## issue 本文フォーマット

```markdown
## 概要
<何をするか / 何が問題か を 2〜3 行で説明>

## 背景・目的
<なぜ必要か、どのユーザー課題を解決するか>

## タスク
- [ ] <具体的な作業 1>
- [ ] <具体的な作業 2>
- [ ] <具体的な作業 3>

## 完了条件
- <この issue が「完了」とみなされる条件>
```

## ラベル候補

| ラベル | 用途 |
|---|---|
| `feature` | 新機能の追加 |
| `bug` | バグ修正 |
| `chore` | リファクタリング・依存更新・設定変更 |
| `docs` | ドキュメントのみの変更 |
| `frontend` | フロントエンドに関する変更 |
| `backend` | バックエンドに関する変更 |
| `infra` | インフラ・CI/CD に関する変更 |

## 実行手順

1. **ユーザー入力を解析**: `$ARGUMENTS` から issue の内容を把握する。情報が不足している場合は確認する。

2. **issue 内容を下書き**:
   - タイトル・本文（概要・背景・タスク・完了条件）・ラベルを作成
   - タスクは具体的な作業単位に分解する

3. **下書きをユーザーに表示し確認を求める**。OKをもらってから作成する。

4. **issue を作成**:
   ```bash
   gh issue create \
     --title "<タイトル>" \
     --label "<ラベル1>,<ラベル2>" \
     --body "$(cat <<'EOF'
   ## 概要
   <概要>

   ## 背景・目的
   <背景>

   ## タスク
   - [ ] <作業1>
   - [ ] <作業2>

   ## 完了条件
   - <完了条件>
   EOF
   )"
   ```

5. **issue の URL をユーザーに返す**

## 重要な注意事項

- ユーザー確認なしに issue を作成しない
- タスクは実装者がすぐ着手できる粒度に分解する
- ラベルが GitHub リポジトリに存在しない場合は `--label` を省略し、作成後にユーザーが手動で付与できるよう URL を案内する
- **issue のタイトルと本文は必ず日本語で作成すること**
