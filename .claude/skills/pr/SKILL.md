---
name: pr
description: サマリー、テストプラン付きでGitHub プルリクエストを作成
---

ユーザー入力はエージェントから直接提供されるか、コマンド引数として提供される可能性があります - プロンプトを進める前に必ず考慮してください（空でない場合）。

ユーザー入力:

$ARGUMENTS

以下の規約に従ってGitHub プルリクエストを作成してください:

## PRタイトルフォーマット

**フォーマット**: `<説明>`

- 説明は簡潔で命令法であること
- 72文字以下に保つ

### 例

- `ローカル開発環境でのGitHub OAuthログインを修正`
- `ユーザープロフィールページを追加`
- `インフラストラクチャを Cloud Run に移行`

## PR本文フォーマット

```markdown
## Summary
<主要な変更の箇条書き — 何が行われ、なぜ行われたか>

## Test plan
<検証手順のチェックリスト>

```

## 実行手順

1. **現在の状況確認**:
   - `git status` を実行してコミットされていない変更を確認
   - **ベースブランチは基本的に `develop` を使用する。`develop` ブランチが存在しない場合のみ `main` を使用する。**
   - `git log` と `git diff <ベースブランチ>...HEAD` を実行してこのブランチの全コミットを理解
   - ブランチがリモートにプッシュされているかチェック

2. **すべての変更を解析**:
   - ブランチ上の全コミットをレビュー（最新のみではない）
   - 変更の全範囲を理解
   - 関連する変更をサマリーの箇条書きにグループ化

3. **PR内容を下書き**:
   - **タイトル**: `<簡潔な説明>`
   - **サマリー**: すべての変更をカバーする箇条書き、何となぜに焦点
   - **テストプラン**: 変更が機能することを検証する方法のチェックリスト（`- [ ]`形式）

4. **下書きをユーザーに表示**し、作成前に確認を求める。下書きはテストプラン項目に`- [ ]`チェックボックス形式を必ず使用すること。

5. **プッシュしてPR作成**:
   - 必要に応じて `-u` フラグでリモートにプッシュ
   - heredocフォーマットで `gh pr create` を使用して検出されたベースブランチを対象にPRを作成:
   ```bash
   gh pr create --base <検出されたベースブランチ> --title "<タイトル>" --body "$(cat <<'EOF'
   ## Summary
   - <変更 1>
   - <変更 2>

   ## Test plan
   - [ ] <検証手順 1>
   - [ ] <検証手順 2>

   EOF
   )"
   ```

6. **PR URLをユーザーに返す**

## 重要な注意事項

- 常にブランチの全コミットを解析、最新のみではない
- サマリーは「何となぜ」に焦点、「どのように」ではない
- テストプランは実行可能で具体的であること
- ユーザー確認なしに強制プッシュしない
- コミットされていない変更がある場合、最初にコミットするかユーザーに尋ねる
- **PRのタイトルと本文（Summary、Test plan）は必ず日本語で作成すること**
